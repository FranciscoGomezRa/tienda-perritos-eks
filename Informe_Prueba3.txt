 IE1 — CONFIGURACIÓN DEL CLÚSTER AWS (VPC, subredes, SG, IAM, nodos)
--------------------------------------------------------------------------------
- VPC: tienda-vpc (10.0.0.0/16) con 6 subredes en 2 AZ (us-east-1a / us-east-1b):
  2 públicas + 4 privadas. NAT Gateway para la salida a internet de las privadas.
- Rol IAM utilizado: LabRole (en Learner Lab no se pueden crear roles nuevos).
- Node group: tipo de instancia t3.medium, desired/min/max = 2 / 2 / 4, en subredes PRIVADAS.

  PERMISOS IAM (políticas ya incluidas en LabRole)
  ------------------------------------------------
  No fue necesario crear ni adjuntar políticas: AWS Academy entrega LabRole con un
  conjunto amplio de políticas administradas que ya cubren lo que EKS necesita (y
  además Learner Lab bloquea la edición de roles IAM). El mismo LabRole se reutiliza
  para el plano de control del clúster y para los nodos. Las 3 políticas clave para
  que los nodos worker operen son:

    • AmazonEKSWorkerNodePolicy — permite que la instancia EC2 se registre y funcione
      como nodo del clúster EKS (comunicación del kubelet con el plano de control).
    • AmazonEKS_CNI_Policy — habilita el plugin de red de EKS (Amazon VPC CNI) para
      asignar IPs de la VPC a los pods y gestionar las interfaces de red (ENI).
    • AmazonEC2ContainerRegistryReadOnly — da permiso de SOLO LECTURA sobre ECR, es
      decir, autoriza el "docker pull" de nuestras imágenes (tienda-frontend/backend/db)
      desde el registro privado hacia los nodos.

  Nota: el rol del PLANO DE CONTROL no hace pull de imágenes (eso ocurre en los nodos);
  su política relevante es AmazonEKSClusterPolicy, también incluida en LabRole.

  JUSTIFICACIÓN DEL DISEÑO DE RED (elegido: patrón de producción)
  ---------------------------------------------------------------
  Se evaluaron dos topologías:

  • Camino A (lab simplificado): nodos en subredes PÚBLICAS con auto-assign de IP
    pública; sin NAT. Ventaja: cero costo de NAT y menos componentes. Desventaja:
    los nodos quedan expuestos con IP pública (mayor superficie de ataque) y las
    4 subredes privadas quedarían casi sin uso.

  • Camino B (ELEGIDO, producción): nodos en subredes PRIVADAS (sin IP pública),
    un NAT Gateway en subred pública para su salida a internet, y el Load Balancer
    en las subredes públicas. Por qué lo elegimos:
      - Seguridad: ningún nodo es accesible directamente desde internet; el único
        punto de entrada es el ELB.
      - Defensa en capas: aunque un pod sea comprometido, el nodo no está expuesto.
      - Coherencia: las 6 subredes cumplen un rol real (públicas=ELB+NAT, privadas=nodos).
      - Es la arquitectura recomendada por AWS para EKS en entornos reales.
    Costo asumido: el NAT consume créditos; se mitigó usando 1 solo NAT y apagando
    el lab al terminar. En producción se usaría 1 NAT por AZ para alta disponibilidad.

  Aclaración técnica: auto-assign de IP pública (en subredes públicas) y NAT NO son
  redundantes: actúan sobre capas distintas. Sería redundante solo si los nodos
  estuvieran en públicas (con IP) y además se enrutaran por NAT.

- Por qué 2 AZ: alta disponibilidad (si una zona de AWS falla, la otra sigue).
- Por qué EKS: orquestación gestionada (auto-recuperación, escalado, rolling updates).

 BITÁCORA DE MEJORAS Y DECISIONES (se actualiza durante todo el proyecto)

Lista central de mejoras aplicadas al proyecto base del profesor. Útil para la
defensa y para el criterio de "optimización / mejora continua".

[Estructura del repositorio]
- Se unificó a un MONOREPO con la app en la raíz (los 3 repos entregados eran
  idénticos byte a byte; se usó 3.6.3). Así GitHub Actions detecta el workflow.
- Material del profesor archivado en docs_profesor/.

[Correcciones de código]

  inválida; corregido a "tienda-db".

[Seguridad / Secrets — IE5]
- Se quitaron credenciales horneadas en db/Dockerfile (la password va por runtime).
- Se eliminó k8s/mysql-secret.yaml (password en base64 versionada) → ahora el Secret
  se GENERA en el pipeline desde GitHub Secrets; queda mysql-secret.example.yaml.
- Variables sensibles en .env (en .gitignore) + .env.example de referencia.
- Secrets en un GitHub Environment "production" (scope, reglas de protección,
  trazabilidad) en vez de sueltos en el repo.

[Despliegue / Red — IE1, IE2]
- frontend-service: se quitaron anotaciones del AWS LB Controller (no está en
  Learner Lab) → Classic ELB con type: LoadBalancer (URL pública).
- Diseño de red de PRODUCCIÓN (Camino B): 2 subredes públicas + 4 privadas en 2 AZ,
  NAT Gateway, nodos worker en privadas (sin IP pública), ELB en públicas.
  Etiquetado de subredes (role/elb e internal-elb) para el balanceo.

[CI/CD — IE4]
- Pipeline mejorado: usa el Environment "production"; genera el Secret de MySQL;
  despliega también la imagen de la DB (set image + rollout); instala Metrics
  Server (para HPA); etiqueta imágenes con SHA corto + latest.
